# Module: money_laundering

## Purpose
Processes all active LaunderingOperation records each tick: transfers criminal revenue from the informal market cash pool to formal market cash through one of six laundering methods (structuring, shell_company_chain, real_estate, trade_invoice, crypto_mixing, cash_commingling), deducting conversion losses, generating method-specific evidence tokens, and updating operation progress. Additionally runs the monthly FIU pattern analysis system that detects suspicious financial activity across provinces and feeds evidence tokens to LE NPCs.

Laundering is architecturally a ledger transfer between market layers. The intermediate steps (shell company chain, bank deposits, real estate transactions) are modelled as the evidence generation profile of the method, not as a full transaction simulation. The evidence is what matters; the accounting is the consequence.

## Inputs (from WorldState)
- `laundering_operations` — all LaunderingOperation records: id, actor_id, method, dirty_amount, laundered_so_far, launder_rate_per_tick, conversion_loss_rate, started_tick, destination_business_id, shell_chain_business_ids, mixer_node_id, evidence_generated_total, paused, completed
- `npc_businesses` — NPCBusiness records for shell companies (criminal_sector == false, owner_id == criminal actor), front businesses (cash_commingling capacity), and destination businesses (clean cash credit)
- `mixer_nodes` — MixerNode records: capacity_per_tick, fee_rate, traceability, operator_npc_id, fiu_flagged
- `player` — PlayerCharacter: informal_cash (debited for player-initiated operations), wealth (credited for direct-to-player destinations)
- `significant_npcs` — NPC records for shell company NPCs (flip risk), banker NPCs (FIU reporting), LE NPCs (evidence delivery target)
- `evidence_pool` — existing evidence tokens for FIU pattern analysis context
- `regional_markets` — market data for revenue/traffic discrepancy detection in commingling analysis
- `provinces` — province list for FIU monthly analysis dispatch
- `current_tick` — for operation progress tracking and FIU monthly scheduling
- `config.laundering.*` — all 16 named constants from simulation_config.json -> laundering

## Outputs (to DeltaBuffer)
- `LaunderingOperationDelta.laundered_so_far` — additive: transfer_this_tick added to running total
- `LaunderingOperationDelta.completed` — set to true when laundered_so_far >= dirty_amount
- `LaunderingOperationDelta.evidence_generated_total` — running sum incremented per new evidence token
- `PlayerDelta.informal_cash` — deducted by transfer_this_tick for player-initiated operations
- `PlayerDelta.wealth` — credited by clean_amount when destination_business_id == 0
- `BusinessDelta.cash_delta` — credited to destination_business_id by clean_amount; debited from criminal_org.cash for NPC-initiated operations
- `CriminalOrgDelta.cash` — debited by transfer_this_tick for NPC criminal org operations
- `EvidenceTokenDelta` — new financial EvidenceToken per method-specific generation rate: structuring (1 per structuring_token_interval ticks), shell_company_chain (1 per chain_node per 30 ticks), real_estate (1 on transaction), trade_invoice (1 per 20 ticks, documentary type), crypto_mixing (investigator-skill-scaled), cash_commingling (1 per 15 ticks, testimonial risk)
- `InvestigatorMeterDelta.fill_rate` — additive fill from FIU pattern results: token.actionability * config.laundering.fiu_meter_fill_scale
- `FIUPatternResultDelta` — monthly pattern analysis outputs: target_actor_id, suspicion_score, inferred_method

## Preconditions
- Criminal operations module has completed; CriminalOrganization.cash and player.informal_cash reflect current criminal revenue.
- Evidence pool contains all tokens generated by preceding modules this tick.
- Shell company chain validation has passed at operation creation: all shell_chain_business_ids have criminal_sector == false and valid owner_id chain.
- MixerNode records are current (traceability, fiu_flagged status).

## Postconditions
- Every active (not paused, not completed) LaunderingOperation has been processed exactly once this tick.
- informal_cash (player) or criminal_org.cash (NPC) decreased by transfer_this_tick for each active operation.
- Clean cash credited to destination at rate: transfer_this_tick * (1.0 - conversion_loss_rate).
- Operations where laundered_so_far >= dirty_amount are marked completed.
- Evidence tokens generated per method-specific profiles, attributed to the operating actor.
- Monthly FIU analysis (when current_tick aligns with monthly schedule) has scanned all provinces for structuring patterns, shell company round-trips, and revenue/traffic discrepancies.

## Invariants
- Transfer per tick: `transfer_this_tick = min(launder_rate_per_tick, dirty_amount - laundered_so_far)`. Never exceeds remaining dirty amount.
- Clean amount: `clean_amount = transfer_this_tick * (1.0 - conversion_loss_rate)`. Conversion loss is always deducted.
- Shell chain constraint: `len(shell_chain_business_ids) <= config.laundering.max_chain_depth` (default 5). All chain businesses must have criminal_sector == false. Cross-province chains require ProvinceLink adjacency. V1 chains are intra-nation only.
- Shell chain rate: each chain node multiplies throughput AND evidence generation linearly. 3-node chain launders 3x faster but generates 3x evidence tokens.
- Cash commingling capacity: bounded by front business legitimate revenue * config.laundering.commingle_capacity_fraction. Cannot inject more than business cash flow plausibly explains.
- Crypto evidence formula: `evidence_probability_per_tick = launder_rate_per_tick * mixer.traceability * max(le_npc.skill(Intelligence) for targeting LE NPCs) / config.laundering.crypto_evidence_skill_divisor`.
- FIU structuring detection: triggers when sub-threshold deposit count > config.laundering.structuring_deposit_count_threshold within 30 ticks.
- FIU suspicion results above config.laundering.fiu_token_threshold (0.35) generate financial EvidenceTokens.
- NPC criminal orgs: total launder_rate_per_tick across all active operations cannot exceed config.laundering.org_capacity_multiplier * org.cash / TICKS_PER_QUARTER.
- NPC orgs use shell_company_chain method by default; do not use crypto or real_estate in V1.
- Real estate laundering gated behind PropertyListing struct availability (BLOCKER-B dependency).
- Floating-point accumulations use canonical sort order (operation_id ascending) for deterministic summation.
- Same seed + same inputs = identical laundering output regardless of core count.
- All random draws (crypto evidence probability, FIU detection) through `DeterministicRNG`.

## Failure Modes
- Shell company NPC flipped (InformantStatus::cooperating): chain is compromised. Log the break point; evidence tokens generated against criminal actor immediately. Operation paused automatically.
- MixerNode fiu_flagged == true: crypto operations using this node generate evidence immediately each tick. Log diagnostic.
- Destination business bankrupt or seized: clean cash has no valid target. Operation paused; player/NPC must reassign destination. Log warning.
- NaN or negative transfer amount from floating-point edge case: clamp to 0.0, log diagnostic.
- Player informal_cash insufficient for transfer_this_tick: clamp transfer to available informal_cash. Operation completes with partial final transfer.

## Performance Contract
- Sequential execution (not province-parallel); laundering operations are actor-level, not province-level.
- Target: < 10ms total for all active laundering operations (~20-50 operations typical) plus monthly FIU analysis.
- FIU monthly analysis: < 20ms across all 6 provinces (runs once per 30 ticks).
- Must not exceed tick budget share that would push total tick above 200ms target.

## Dependencies
- runs_after: ["criminal_operations", "evidence"]
- runs_before: ["investigator_engine"]

## Test Scenarios
- `test_structuring_transfers_and_generates_evidence`: Create a structuring operation with dirty_amount 10000, rate 500/tick. Run 7 ticks. Verify laundered_so_far = 3500, one financial EvidenceToken generated (structuring_token_interval = 7).
- `test_shell_chain_rate_scales_with_depth`: Create two shell_company_chain operations: 1-node chain and 3-node chain. Verify 3-node chain launders at 3x rate (rate_chain_per_node * chain_length) and generates 3x evidence tokens.
- `test_conversion_loss_deducted`: Create operation with conversion_loss_rate 0.05, transfer 1000. Verify destination receives 950, not 1000.
- `test_crypto_evidence_scales_with_investigator_skill`: Create crypto_mixing operation. Two scenarios: LE NPC with Intelligence skill 2.0 vs 8.0. Verify evidence_probability_per_tick is 4x higher for skilled investigator.
- `test_commingling_capped_by_business_revenue`: Front business with revenue_per_tick 5000. Verify commingling rate capped at min(5000 * 0.40, config.laundering.rate_commingle_max).
- `test_fiu_detects_structuring_pattern`: Actor makes 10 sub-threshold deposits in 30 ticks (above threshold of 8). Verify FIU generates FIUPatternResult with suspicion_score and financial EvidenceToken.
- `test_fiu_detects_shell_round_trip`: Three shell companies with same owner_id where cash_in approximately equals cash_out in 90 ticks. Verify FIU generates shell_company_chain pattern result with suspicion_score 0.6.
- `test_operation_completes_when_fully_laundered`: Operation with dirty_amount 5000, rate 500/tick. Run 10 ticks. Verify completed == true and laundered_so_far >= dirty_amount.
- `test_paused_operation_not_processed`: Set operation.paused = true. Verify no transfer occurs, no evidence generated, laundered_so_far unchanged.
- `test_npc_org_capacity_limit`: NPC criminal org with cash 100000. Verify total launder_rate_per_tick across all operations does not exceed 0.25 * 100000 / 90.
- `test_shell_company_flip_compromises_chain`: Mark a shell company NPC as InformantStatus::cooperating. Verify operation is paused and high-actionability documentary EvidenceToken generated against the criminal actor.
- `test_determinism_across_runs`: Run 100 ticks of laundering with multiple concurrent operations using fixed seed. Verify bit-identical output across two runs.
